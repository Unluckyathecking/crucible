## 2026-05-16 - StatusRecorder overwriting status issue
**Learning:** `http.ResponseWriter.WriteHeader` honors only the first call, but wrapping loggers (`StatusRecorder`) might overwrite their internal state on multiple calls if not specifically preventing it, causing access logs and metrics to log a wrong status code. Be sure to respect 1xx informational status codes, which do not finalize the header response.
**Action:** When implementing an HTTP middleware logger tracking the status, add a `wroteHeader` boolean flag. Only record `code >= 200` to lock the flag, and implicitly lock it on any `Write` calls. Crucially, in `Write`, if `!wroteHeader`, set the status to `http.StatusOK` (200) to ensure the status correctly reflects the implicit 200 sent by the standard library. Otherwise, `Status` might remain 0 if constructed via a struct literal rather than a constructor.
## 2026-05-26 - Cache auth failures to prevent DB fallthrough DoS
**Learning:** When caching database lookups that involve subsequent validation (like API key hashing), if validation fails and the code doesn't explicitly return early or cache the failure, it causes a "cache fallthrough" vulnerability. Attackers can spam invalid hashes for a valid prefix, continually missing the intended cache hit path, bypassing the cache entirely, and hitting the database on every request, leading to database connection pool exhaustion and DoS. However, creating a special "negative cache" entry by inserting a blank/invalid hash into the cache poisons it, causing legitimate requests with the *correct* hash to fail.
**Action:** Always populate the cache on the cold path with the valid record *before* validation so that future invalid requests hit the cache and fail verification, while future legitimate requests hit the cache, pass verification, and are granted access. Never cache unverified/invalid input. On cache hits, terminate early if validation fails.
